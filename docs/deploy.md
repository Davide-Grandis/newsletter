# Deployment Runbook

End-to-end deployment of the newsletter pipeline to a fresh Cloudflare zone.
All commands assume `cwd = repo root` unless noted.

Replace `yourdomain.com` throughout this runbook with your actual Cloudflare zone
domain, and `your@email.com` with the author email address you'll use for the
first test send.

## 0. Prerequisites

Done once per machine.

```bash
# Node 20+
node --version

# Wrangler 3
npx wrangler --version

# Authenticate to the target Cloudflare account.
# This opens a browser; pick the account that owns your zone.
npx wrangler login

# Confirm.
npx wrangler whoami
```

If you have multiple accounts, export the right one for the rest of the
session so wrangler doesn't prompt:

```bash
export CLOUDFLARE_ACCOUNT_ID=<your-account-id>
```

Install repo dependencies (workers + SPA):

```bash
npm install
(cd web && npm install)
```

## 1. Provision Cloudflare resources

Each command prints an ID. **Copy them**; you'll paste them into the
`wrangler.toml` files in step 2.

### 1.1 D1 database

```bash
npx wrangler d1 create newsletter_db
# Output:
# database_id = "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx"
```

Apply the schema (one-shot bootstrap — required so the admin worker has
tables to talk to):

```bash
npx wrangler d1 execute newsletter_db --remote --file=db/schema.sql
```

Everything else (seeding the first author, adding subscribers, etc.) is
done through the admin GUI once it is deployed and gated by Access — see
section 9.1.

### 1.2 Queues

```bash
npx wrangler queues create newsletter-queue
npx wrangler queues create newsletter-dlq
```

(No IDs to copy; bindings reference queues by name.)

### 1.3 R2 bucket

```bash
npx wrangler r2 bucket create newsletter-archive
```

## 2. Patch `database_id` into all six wrangler configs

Replace `REPLACE_WITH_D1_ID` with the UUID from step 1.1 across all six
worker configs:

```bash
DB_ID="paste-the-uuid-here"
for f in workers/{ingest,consumer,tracker,bounce,cleanup,admin}/wrangler.toml; do
  sed -i.bak "s/REPLACE_WITH_D1_ID/$DB_ID/" "$f" && rm "$f.bak"
done

grep database_id workers/*/wrangler.toml   # confirm
```

## 3. Configure DNS, Email Routing, and Email Sending in the dashboard

These steps are dashboard-only; wrangler can't drive them.

### 3.1 Email Routing (inbound)

1. Cloudflare dashboard → your zone → **Email** → **Email
   Routing** → enable.
2. Cloudflare will add MX + SPF records automatically.
3. Routes will be added in step 6 once the workers exist.

### 3.2 Email Sending (outbound, beta)

1. Same panel → **Email Sending** → enable.
2. Add the DKIM TXT record it generates.
3. Wait until DKIM validation goes green (usually <5 min).
4. The `[[send_email]]` binding on `consumer/wrangler.toml` will pick up
   automatically once deployed in the same account.

### 3.3 DNS for the tracker

**Skip this — the tracker's `custom_domain = true` route creates the DNS
record automatically when step 7 deploys.** Manually adding an A or CNAME
for `track.yourdomain.com` will cause `wrangler deploy` to fail with
"Hostname already has externally managed DNS records".

## 4. Set worker secrets

Each `wrangler secret put` opens a prompt; paste the value and Enter.

```bash
# Per-worker secrets
for w in tracker consumer; do
  (cd workers/$w && \
    npx wrangler secret put LINK_SIGNING_KEY && \
    npx wrangler secret put ATTACHMENT_SIGNING_KEY)
done

# The admin worker has no secret — it is gated by Cloudflare Access.
# See section 8 below.
```

Generate strong values with:

```bash
openssl rand -base64 48      # for *_SIGNING_KEY
```

The two signing keys MUST be identical between the consumer (which signs)
and the tracker (which verifies) — paste the exact same value to both.

## 5. Build SPA + deploy workers

The admin worker serves the SPA from `workers/admin/public/`, which is
generated by `npm run build:web`.

```bash
# One command builds the SPA and deploys all six workers in dependency order.
npm run deploy:all
```

If you'd rather deploy one at a time (e.g. to debug):

```bash
npm run build:web
(cd workers/ingest    && npx wrangler deploy)
(cd workers/consumer  && npx wrangler deploy)
(cd workers/tracker   && npx wrangler deploy)
(cd workers/bounce    && npx wrangler deploy)
(cd workers/cleanup   && npx wrangler deploy)
(cd workers/admin     && npx wrangler deploy)
```

After each deploy, wrangler prints the worker's `*.workers.dev` URL. Save
the **admin** and **tracker** URLs; you'll need them.

## 6. Wire Email Routing rules to the deployed workers

Dashboard → Email Routing → **Routes**.

| Match                                | Action                              |
| ------------------------------------ | ----------------------------------- |
| `newsletter@yourdomain.com`       | Send to Worker → `newsletter-ingest`|
| `bounce+*@yourdomain.com` (custom)| Send to Worker → `newsletter-bounce`|

The `bounce+*` pattern needs a **catch-all** rule with action "send to
worker", because Email Routing's UI doesn't accept `+*` literally; the
catch-all worker handler in `bounce/src/index.ts` then filters by prefix.

## 7. Bind the tracker to its custom hostname

After step 5 prints the tracker's `*.workers.dev` URL, edit
`workers/tracker/wrangler.toml`:

```toml
routes = [{ pattern = "track.yourdomain.com/*", custom_domain = true }]
```

Then redeploy:

```bash
(cd workers/tracker && npx wrangler deploy)
```

## 8. Put the admin GUI behind Cloudflare Access (REQUIRED)

The admin worker has **no built-in authentication**. It only checks for
the `Cf-Access-Authenticated-User-Email` header that Cloudflare Access
injects after a successful login. Without an Access application in front
the API will return `401` on every `/api/*` call.

1. Zero Trust → **Access** → **Applications** → Add → Self-hosted.
2. Application domain: the admin worker's URL.
3. Identity providers: pick one (Google, OTP, …).
4. Policy: `Allow` for `Emails: your@email.com` (or a group).

Once saved, the GUI header auto-populates the user's name and the Logout
button hits `/cdn-cgi/access/logout`.

## 9. Smoke test

### 9.1 Bootstrap data via the admin GUI

Open the admin URL in a browser (Access will prompt for SSO):

```
https://newsletter-admin.<your-subdomain>.workers.dev/
```

Then, using the GUI:

1. **Authors** page → add `your@email.com` (or whichever address
   you'll send the test from). The ingest worker rejects every inbound
   email until at least one row exists here.
2. **Subscribers** page → add at least one subscriber so the test
   campaign has somewhere to deliver.

### 9.2 Trigger an end-to-end send

```bash
# Send a test newsletter to ingest, FROM the authorized author address.
echo "Hello world" | mail -s "Test campaign" newsletter@yourdomain.com

# Watch logs.
npx wrangler tail newsletter-ingest
npx wrangler tail newsletter-consumer
```

If ingest logs `Sender not authorized`, the inbound `From:` doesn't match
any row in the `authors` table — add it via the GUI's *Authors* page or
`POST /api/authors`.

You should see: ingest parses → consumer picks up the queue batch → SEND_EMAIL
delivers → the test subscriber receives the message with tracking pixel,
signed click links, and an unsubscribe header.

The Dashboard at `$ADMIN_URL/` will show the campaign in real time.

## 10. (Later) Enable warmup

When you're ready to start sending to real lists, edit both
`workers/consumer/wrangler.toml` and `workers/admin/wrangler.toml`:

```toml
WARMUP_START_DATE = "2026-04-28"   # today's UTC date
```

Redeploy both:

```bash
(cd workers/consumer && npx wrangler deploy)
(cd workers/admin    && npx wrangler deploy)
```

The Dashboard's Quota panel will start showing daily/weekly caps.

## Rollback

```bash
# Revert a worker to the previous version.
(cd workers/<name> && npx wrangler rollback)
```

D1, R2, and Queues are not touched by `wrangler deploy` — only worker code
and bindings change.
