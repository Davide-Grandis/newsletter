# Open & Click Tracking — Complete Reference

This document describes how the newsletter pipeline instruments outgoing email
to record opens and clicks: the mechanism, the security model, the data stored,
what each worker does, and how to configure or disable tracking.

---

## Glossary

| Term | Meaning |
| --- | --- |
| **Open tracking** | Detecting when a subscriber opens (renders) an email by embedding an invisible 1×1 pixel image. When the mail client loads the image, the tracker worker is called and records an `open` event. |
| **Click tracking** | Detecting when a subscriber clicks a link by replacing the original URL with a signed redirect through the tracker worker. The worker records a `click` event and immediately redirects to the original destination. |
| **Tracking pixel** | A 1×1 transparent GIF appended to the HTML body of every email. Its URL encodes the campaign and subscriber IDs; the tracker logs the request as an `open`. |
| **Signed URL** | A URL that includes an HMAC-SHA-256 authentication code over the encoded parameters. The tracker worker verifies the code before acting, preventing forgery or enumeration. |
| **HMAC-SHA-256** | Keyed-hash message authentication code using SHA-256. A shared secret key is used to produce and verify a short code that proves a URL was generated by the system. |
| **`LINK_SIGNING_KEY`** | A shared secret (Worker Secret) used to sign and verify click redirect and open-pixel URLs. Must be identical on the consumer and tracker workers. |
| **`ATTACHMENT_SIGNING_KEY`** | A separate secret used to sign attachment download URLs. See `docs/attachments.md`. |
| **Instrumentation** | The process of rewriting all `href="https://…"` links in the HTML body and appending the tracking pixel; performed by `instrumentHtml()` in `shared/tracking.ts`. |
| **Base64url** | URL-safe Base64 encoding (replaces `+` with `-`, `/` with `_`, strips `=` padding). Used for HMAC signatures in URLs. |
| **Consumer worker** | `workers/consumer` — builds and sends per-recipient emails; applies `instrumentHtml()` to each HTML body when tracking is enabled. |
| **Tracker worker** | `workers/tracker` — handles all tracking requests: open pixels, click redirects, attachment downloads, unsubscribe, and public signup. Lives at `track.<domain>`. |
| **`events` table** | D1 table recording every open, click, download, unsubscribe, bounce, and complaint event with timestamp, IP, and user-agent. |
| **Tracking base URL** | The root URL of the tracker worker (`TRACKING_BASE_URL`, e.g. `https://track.eneanewsletter.it`). Used as the prefix for all tracking and unsubscribe URLs. |

## 1. Data model

All tracking signals are written to the D1 `events` table (`db/schema.sql`):

| Column | Purpose |
| --- | --- |
| `id` | Auto-increment primary key. |
| `campaign_id` | Campaign the event belongs to (`ON DELETE CASCADE`). `NULL` for unsubscribe events not linked to a campaign. |
| `subscriber_id` | Subscriber who triggered the event (`ON DELETE SET NULL`). |
| `type` | One of `open`, `click`, `download`, `unsubscribe`, `bounce`, `complaint`. |
| `attachment_id` | For `download` events: the attachment that was downloaded. |
| `url` | For `click` events: the destination URL. For `download`: the filename. |
| `ts` | UTC timestamp of the event. |
| `ua` | `User-Agent` header from the request. |
| `ip` | `CF-Connecting-IP` of the request. |

Events are surfaced on the admin console's **Analytics** page, grouped by
campaign.

## 2. How links are instrumented

At send time, the **consumer worker** calls `instrumentHtml()` from
`shared/tracking.ts` on the HTML body of each email (once per subscriber, since
the URLs are subscriber-specific):

```ts
instrumentHtml(html, TRACKING_BASE_URL, LINK_SIGNING_KEY, campaignId, subscriberId)
```

The function:

1. **Rewrites every `href="https://…"` link** in the HTML to a signed click URL:

   ```
   https://track.<domain>/c/<campaignId>/<subscriberId>?u=<encoded-target>&sig=<hmac>
   ```

   - Only `http://` and `https://` URLs are rewritten. Mailto, relative, and
     other scheme links are left unchanged.
   - The HMAC payload is: `c|<campaignId>|<subscriberId>|<url-encoded-target>`.
     The target is encoded *before* signing so the signature is computed over
     the encoded form (matching what the tracker receives).

2. **Appends a tracking pixel** at the very end of the HTML body:

   ```html
   <img src="https://track.<domain>/o/<campaignId>/<subscriberId>.gif"
        width="1" height="1" alt="" style="display:none">
   ```

The footer is appended **after** `instrumentHtml()` so unsubscribe links in the
footer are never click-rewritten.

Link-mode attachment download links (`/a/…`) are also excluded from click
rewriting — they are generated by `signDownloadUrl()` and already point at the
tracker worker.

---

## 3. Open events (`/o/:campaign/:sub.gif`)

When a subscriber's mail client renders the HTML body, it fetches the pixel:

```
GET https://track.<domain>/o/<campaignId>/<subscriberId>.gif
```

The tracker worker:

1. Extracts `campaignId` and `subscriberId` from the path (the `.gif` suffix is
   stripped from the subscriber segment).
2. Fires `logEvent(env, 'open', campaignId, sub, req, null)` via
   `ctx.waitUntil()` — async, does not delay the response.
3. Returns the **transparent GIF** (43-byte 1×1 pixel):

   ```
   Content-Type: image/gif
   Cache-Control: no-store, max-age=0
   ```

No authentication is required for the pixel — the URL itself is subscriber-
specific but not secret. Double-loads (e.g. from a preview pane re-rendering)
create multiple `open` events; the Analytics page de-duplicates them by
counting distinct opens per campaign-subscriber pair.

> **Important:** open tracking requires the recipient's mail client to load
> remote images. Many clients (Gmail, Outlook, Apple Mail) block or proxy image
> loading, so open counts are a lower-bound estimate, not an exact figure.

---

## 4. Click events (`/c/:campaign/:sub`)

When a subscriber clicks a tracked link:

```
GET https://track.<domain>/c/<campaignId>/<subscriberId>?u=<encoded-target>&sig=<hmac>
```

The tracker worker:

1. Reads the raw (still URL-encoded) `u` parameter directly from the query
   string — **not** via `URLSearchParams.get()`, which would decode it. This
   preserves the exact bytes the HMAC was computed over.
2. Verifies: `HMAC-SHA-256(LINK_SIGNING_KEY, "c|<campaignId>|<sub>|<encoded-u>") == sig`.
   - Mismatch → `403 Forbidden`.
3. Decodes `u` and fires `logEvent(env, 'click', …, target)`.
4. Returns `302 Redirect` to the original destination URL.

The redirect is instantaneous from the subscriber's perspective. The log write
is fired via `ctx.waitUntil()` — it does not block the redirect.

---

## 5. Download events (`/a/:campaign/:sub/:attId`)

See `docs/attachments.md` §5.2 for full detail. In summary:

```
GET https://track.<domain>/a/<campaignId>/<subscriberId>/<attachmentId>?sig=<hmac>
```

Tracker verifies `HMAC-SHA-256(ATTACHMENT_SIGNING_KEY, "a|…|…|…")`, fetches
the file bytes from R2, logs `type='download'`, and streams the file.

---

## 6. What is NOT tracked

- **Plain-text part**: the text version of each email never contains a pixel or
  rewritten links. Subscribers who read the text-only version cannot be tracked.
- **Footer unsubscribe links**: explicitly excluded from click rewriting
  (`instrumentHtml` runs before the footer is appended).
- **Inline image `cid:` references**: these are not `https://` links and are
  never rewritten.
- **Non-HTML links** (`mailto:`, relative URLs, etc.): the regex requires
  `https?://`, so these are left unchanged.

---

## 7. Disabling tracking

Tracking is **opt-out per deployment** via the `TRACKING_ENABLED` setting.

When `TRACKING_ENABLED = false`:

- `instrumentHtml()` is **not called**. Links are delivered unmodified. No pixel
  is appended.
- Download links for link-mode attachments are **still generated and signed** —
  they use `ATTACHMENT_SIGNING_KEY`, which is independent of `TRACKING_ENABLED`.
- The `events` table receives no `open` or `click` rows for new sends. Analytics
  will show zero opens/clicks going forward.
- Unsubscribe links are unaffected — they are generated separately and always
  present.

Change via **Settings → Tracking & signup → Open & click tracking**.

## 8. Security properties

### HMAC binding

Each signed URL is bound to a specific campaign and subscriber. An HMAC computed
for subscriber A cannot be used for subscriber B (the subscriber ID is in the
signed payload). An HMAC for campaign X cannot be used for campaign Y.

### Constant-time comparison

`verifyHmac()` in `shared/tracking.ts` recomputes the expected HMAC and compares
character-by-character using XOR, accumulating differences. This avoids
timing-oracle attacks that could recover the key via response-time differences.

### Key rotation

If `LINK_SIGNING_KEY` is rotated, all existing tracked URLs in already-sent
emails become invalid (clicks will return 403). Rotate only if the key is
compromised; otherwise keep it stable.

## 9. Configuration

| Setting / Secret | Where | Default | Meaning |
| --- | --- | --- | --- |
| `TRACKING_ENABLED` | D1 settings | `true` | When `false`, links are not rewritten and no pixel is added. |
| `TRACKING_BASE_URL` | D1 settings | `https://track.eneanewsletter.it` | Base URL of the tracker worker — prefix for all tracking, download, and unsubscribe URLs. |
| `LINK_SIGNING_KEY` | Worker Secret (consumer + tracker) | — | HMAC key for click and open-pixel URLs. Must match on both workers. |
| `ATTACHMENT_SIGNING_KEY` | Worker Secret (consumer + tracker) | — | Separate HMAC key for download URLs. Must match on both workers. |

Set secrets with:

```bash
(cd workers/consumer && wrangler secret put LINK_SIGNING_KEY)
(cd workers/tracker  && wrangler secret put LINK_SIGNING_KEY)
(cd workers/consumer && wrangler secret put ATTACHMENT_SIGNING_KEY)
(cd workers/tracker  && wrangler secret put ATTACHMENT_SIGNING_KEY)
```

Generate key values with `openssl rand -base64 48`.
